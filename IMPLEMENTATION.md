# Описание реализации

## Архитектура

Проект реализован с использованием Clean Architecture принципов:

1. **ServerPool.Core** - Доменный слой
   - Модели данных (`Server`, `ServerStatus`)
   - DTOs для API запросов/ответов
   - Интерфейсы сервисов

2. **ServerPool.Infrastructure** - Слой инфраструктуры
   - Entity Framework контекст
   - Реализация сервисов
   - Фоновые задачи (Background Services)

3. **ServerPool.API** - Слой представления
   - REST API контроллеры
   - Конфигурация приложения
   - Swagger документация

4. **ServerPool.Tests** - Тесты
   - Unit тесты для сервисов
   - Unit тесты для контроллеров

## Реализованные функции

### 1. Управление серверами
- ✅ Добавление серверов в пул с указанием характеристик (ОС, память, диск, ядра)
- ✅ Поддержка включенных и выключенных серверов при добавлении

### 2. Поиск серверов
- ✅ Поиск свободных серверов по параметрам:
  - Операционная система (частичное совпадение)
  - Минимальная память
  - Минимальный диск
  - Минимальное количество ядер
- ✅ Возвращает только доступные серверы (Available или готовые PoweringOn)

### 3. Аренда серверов
- ✅ Взятие сервера в аренду с указанием пользователя
- ✅ Автоматическое включение выключенных серверов (переход в PoweringOn)
- ✅ Проверка готовности сервера перед выдачей

### 4. Освобождение серверов
- ✅ Освобождение сервера из аренды
- ✅ Переход в статус Available

### 5. Автоматические процессы
- ✅ **Автоматическое включение**: Серверы в статусе PoweringOn автоматически переходят в Available через 5 минут
- ✅ **Автоматическое отключение**: Серверы автоматически отключаются (Offline) через 20 минут после выдачи

### 6. Проверка готовности
- ✅ API endpoint для проверки готовности сервера к выдаче
- ✅ Возвращает статус и примерное время готовности

## Технические детали

### Обработка конкурентных запросов
Используется `ConcurrentDictionary<Guid, SemaphoreSlim>` для создания блокировок на уровне каждого сервера. Это обеспечивает:
- Thread-safe операции при конкурентных запросах
- Изоляцию операций для разных серверов
- Предотвращение race conditions

### Фоновые задачи
1. **ServerPowerOnService**
   - Проверяет каждые 30 секунд серверы в статусе PoweringOn
   - Переводит в Available, если прошло 5 минут с момента запроса включения

2. **ServerAutoShutdownService**
   - Проверяет каждую минуту выделенные серверы
   - Автоматически отключает серверы через 20 минут после выдачи

### База данных
- Поддержка InMemory базы данных (по умолчанию, для разработки)
- Поддержка SQL Server (для продакшена)
- Entity Framework Core 8.0
- Автоматическое создание схемы при запуске

### Логирование
- Все операции логируются с использованием ILogger
- Уровни логирования: Information, Warning, Error
- Логирование в консоль и Debug output

### Тестирование
- Unit тесты для основных операций сервисов
- Unit тесты для API контроллеров
- Использование Moq для мокирования зависимостей
- Использование FluentAssertions для читаемых assertions
- InMemory база данных для тестов

## Статусы серверов

1. **Available** - Сервер включен и доступен для аренды
2. **Allocated** - Сервер выдан в аренду
3. **PoweringOn** - Сервер включается (5 минут)
4. **Offline** - Сервер выключен

## Потоки данных

### Добавление сервера
```
POST /api/servers
→ ServerService.AddServerAsync()
→ Сохранение в БД
→ Возврат созданного сервера
```

### Поиск серверов
```
POST /api/servers/search
→ ServerService.SearchAvailableServersAsync()
→ Фильтрация по параметрам
→ Возврат списка доступных серверов
```

### Аренда сервера
```
POST /api/servers/allocate
→ ServerService.AllocateServerAsync()
→ Проверка статуса (Offline → PoweringOn, PoweringOn → проверка готовности)
→ Если Available → переход в Allocated
→ Сохранение в БД
→ Возврат выделенного сервера
```

### Освобождение сервера
```
POST /api/servers/{id}/release
→ ServerService.ReleaseServerAsync()
→ Проверка статуса (должен быть Allocated)
→ Переход в Available
→ Сохранение в БД
```
